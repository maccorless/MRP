Concept B · Story-led form, with conditional window-check moderate rethink

Lead with the editorial story (what IOC reviewers actually judge); end with admin data. NOC window is not centre-stage — default is "all NOCs open" so no pre-flight at all. The check only surfaces when it's actually useful.

B.1 · Window-state routing (the smart part)

One cheap server check on page load decides which landing to render. Invitees always bypass — an invite is authorisation, regardless of window state.

State 1 · All NOCs open

Default — common case

No pre-flight. Landing = plain email entry, same as today. Don't tax every applicant with a check that's almost never actionable.

State 2 · Closed except OCOG reopens

Global deadline passed

Default messaging = "window closed". Landing asks NOC up front — only users from an OCOG-reopened NOC get past it. See B.2.

State 3 · All NOCs closed

No reopens

Landing explains the window has closed, offers status-check for existing applicants, and surfaces the invite-only path.

Fast path — invite link

Arrivals with a valid ?token=… invite skip window checks entirely. The invite is the NOC's explicit authorisation; UI should not second-guess it. This is how late extra-time applicants will flow through in practice.

B.2 · Landing page — State 2 (closed-by-default, a few NOCs reopened)

After the global deadline, the majority of applicants hitting /apply should see "closed" messaging. Only users from an OCOG-reopened NOC should be allowed through.

Which direction?

My ranked recommendation:

  1. Concept A as the baseline rebuild — highest ROI, lowest risk, keeps all existing content and the 5-group content model.
  2. Adopt Concept B's window-state routing (3 landing variants + invite fast-path). This is cheap to implement and avoids showing pre-flight UI to the majority of applicants who don't need it.
  3. Concept B's story-first 3-step reorder is a bigger content restructure — worth doing, but plan it as a second phase after A ships.
  4. Concept C is worth prototyping for the mobile variant only — probably not the primary desktop experience.
