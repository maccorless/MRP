# Automated Test Plan — 2026-04-26 PRP Changes

Covers the changes shipped in commit `824faee` ("feat(prp): apply Strategic Plan re-review + Emma 2026-04-24 feedback walkthrough") on 2026-04-26. Goal: get every behavioural change covered by a Vitest integration test or a Playwright smoke E2E before the next stakeholder meeting.

## Test infrastructure recap

- **Unit + integration tests** live in `src/test/`, run with `bun test` (vitest). Tests run against the real Postgres configured in `.env.local`. Use `T_` prefixes for test-data names; clean up in `afterAll`. Helpers are in `src/test/helpers.ts` (`callAction`, `cleanupTestData`, `SESSIONS`, `makeFormData`, `setSession`, `expectRedirect`). See `src/test/uc-noc-direct-entry.test.ts` for the canonical mock-cookie / mock-redirect boilerplate.
- **E2E tests** live in `e2e/smoke/` (functional) and `e2e/a11y/` (accessibility). Run with `bun test:e2e`. Auth state is pre-seeded via `e2e/fixtures/auth.setup.ts`; per-role storage states sit at `e2e/.auth/{noc,ocog,ioc}.json`.
- **Migrations 0027 + 0028** must be applied before tests run — `bun db:migrate`.

## Pre-flight before running this plan

1. `bun db:migrate` — apply `0027_noc_es_category` and `0028_audit_action_noc_pbn_cancel`.
2. `bun db:seed` — refresh seed data (the seed script must populate `noc_es_total` and `enr_requests` rows under `IOC_DIRECT` for the new tests; if it doesn't, add it).
3. `bun test` — confirm pre-existing tests pass before adding new ones (the 7 pre-existing TS errors in `auth-boundaries.test.ts` predate this work and are unrelated).
4. `bun test:e2e` — confirm the smoke suite passes.

## Test plan by change

### 1. Eligibility helper — `.gov` detection

**File to create:** `src/test/lib-eligibility.test.ts` (new).
**Type:** pure unit test, no DB.

**Cases:**
- `isGovernmentEmail("foo@example.gov")` → true.
- `isGovernmentEmail("foo@dept.gov.uk")` → true (with country code).
- `isGovernmentEmail("foo@dept.gov.au")` → true.
- `isGovernmentEmail("foo@example.com")` → false.
- `isGovernmentEmail("foo@gov-example.com")` → false (gov is not the TLD).
- `isGovernmentEmail("foo@government.com")` → false.
- `isGovernmentEmail("")`, `null`, `undefined`, missing `@` → false.
- `ineligibilityFlags({ contactEmail: "foo@dept.gov" })` → returns one flag with `code: "gov_domain"`.
- `ineligibilityFlags({ contactEmail: "foo@example.com", orgEmail: "info@dept.gov" })` → returns the gov flag (any of the three email fields can trigger).
- `ineligibilityFlags({})` with no emails → empty array.

### 2. NOC government soft-warn flag — server enforcement

**File:** extend `src/test/uc-noc-evaluate.test.ts` (or create `uc-noc-eligibility-flag.test.ts` if better grouped).
**Type:** integration; uses real DB.

**Setup:** seed an org with `orgEmail = "press@example.gov"`, an application with `contactEmail = "j.doe@dept.gov.uk"`, status `pending`, NOC = `T_USA`.

**Cases:**
- Calling `approveApplication` with the form data missing `ack_eligibility_flag` redirects to `/admin/noc/{id}?error=eligibility_ack_required`. Application status remains `pending`.
- Same call with `ack_eligibility_flag=1` succeeds: status flips to `approved`, audit row written, redirect to `/admin/noc/queue?success=approved`.
- Application without any `.gov` email needs no ack: succeeds even when `ack_eligibility_flag` is absent.

### 4. NOC allocation hierarchy soft-sort

### 8. Cancel PbN entry action

**File:** new `src/test/uc-noc-pbn-cancel.test.ts`.
**Type:** integration; uses real DB.

**Setup:** seed an EoI-sourced approved org with a `draft` allocation and a Direct-Entry-sourced org with a `draft` allocation, both for NOC `T_USA`.

**Cases (state machine):**
- Cancel an EoI-sourced allocation in `draft`: allocation row deleted; the `applications` row stays at `approved`; audit log row written with `action = "noc_pbn_cancel"`; redirect = `/admin/noc/pbn?success=entry_cancelled`.
- Cancel an EoI-sourced allocation in `noc_submitted`: same as above; allocation row deleted, app stays.
- Cancel a Direct-Entry-sourced allocation in `draft`: allocation row deleted AND `applications.status` flips to `rejected` with `reviewNote` containing the supplied reason.
- Attempt to cancel an `ocog_approved` allocation: redirect to `/admin/noc/pbn?error=cancel_not_allowed`; nothing changes.
- Attempt to cancel a `sent_to_acr` allocation: same `cancel_not_allowed` error.
- Missing `organizationId` form field: redirect to `/admin/noc/pbn?error=missing_org`.
- Wrong `nocCode` (org belongs to a different NOC): redirect to `/admin/noc/pbn?error=alloc_not_found`.

**E2E (light touch):** add to `e2e/smoke/noc-admin.spec.ts` — open PbN, click an org, assert the "Danger zone — Cancel PbN entry" form appears when allocation is editable, and is absent when the allocation is locked.
