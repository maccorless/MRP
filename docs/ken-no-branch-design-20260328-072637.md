## Problem Statement

The IOC's press accreditation process for the Olympic Games is managed through Excel
spreadsheets emailed between the IOC, National Olympic Committees (NOCs), and the
Organising Committee (OCOG). There are 3 parties that suffer:

1. **IOC Media Operations** (worst pain) — they allocate quotas to 206 NOCs but have
   zero visibility into what NOCs do with those quotas. When a media organisation
   disputes their accreditation decision, the IOC cannot see what happened. This is a
   transparency and reputational risk.

2. **NOC press accreditation managers** — they receive Excel templates by email, fill
   them in, send them back. Quota changes trigger re-sends of the entire file. Multiple
   file versions, no audit trail, no deduplication across NOCs.

3. **OCOG ACR staff (LA28)** — they receive submissions from 206 NOCs, manually
   reconcile them, check for duplicates, coordinate corrections. Entirely manual,
   no tooling.

For LA28 2028, the IOC has committed to launching a dedicated Media Registration Portal.
This has been announced to all NOCs. There is no backup plan. The portal must be live
by August 24, 2026.

## Status Quo

NOC coordinator receives an Excel template pre-populated with their quota allocation.
They fill in organisation names, contact details, and category breakdowns. They email it
back to the OCOG. If the IOC revises any quotas, a new version of the template is sent,
and the NOC re-fills it. This creates multiple file versions.

OCOG ACR staff manually process each submission: checking for duplicates across NOCs,
verifying quotas haven't been exceeded, following up on errors. This is entirely manual.

IOC Media Ops has no real-time visibility. They can only see the final approved submissions.

For ENR (non-media rights-holding broadcasters): NOCs collect a "long list" and send it
to the IOC via Media Shuttle or email. IOC reviews and approves. Entirely manual.

## Target Users & Narrowest Wedge

**Primary user:** NOC press accreditation coordinators (~206 NOCs worldwide, range from
large professional teams to one-person volunteer operations). They need:
- A dashboard that shows their territory's applications
- Quota visibility in real time (can't go over their IOC-allocated cap)
- Easy download for their own systems

**Secondary user (buyer/oversight):** IOC Media Ops team (~5-10 people). They need:
- Full visibility across all NOCs — read-only
- Deduplication alerts (same org applying through multiple NOCs)
- Audit trail for dispute resolution

**Narrowest wedge:** Full build per IOC spec is required (IOC has announced the portal
to all NOCs, no partial ship possible). However, the product lens means we build it
generically enough that it's not "LA28 custom software" — it's a multi-event platform
with LA28 as the first event.

**Key scope from IOC spec (v1 vs v1.1):**
1. [v1] Media organisation registration (public form, Expression of Interest phase)
2. [v1] NOC management dashboard (territory-scoped, quota-enforced)
3. [v1] IOC oversight dashboard (all-NOC read-only, deduplication flags)
4. [v1] ACR system integration (data handoff — stubbed in dev, real in prod)
5. [v1.1] Press by Number (quota management, NOC allocation, IOC approval) — ships late Sept 2026 before Oct 5 Press by Number launch
6. [v1.1] ENR workflow (NOC long-list submission, IOC review/approval)
7. [v1.1] Undertaking signature workflow (for accredited ENR organisations)

## Constraints

- **Hard deadline:** April 2026 for testing/systems-ready; August 24, 2026 for live launch.
  **NOTE:** As of this document date (March 28, 2026), April testing is ~3 weeks away.
  This requires immediate team spin-up and scope reduction — not a risk to be deferred.
- **Integration:** Must connect to the ACR system module (the OCOG's accreditation system)
  for the Press by Name phase. ACR API will be stubbed during development.
  **Critical:** The adapter interface must be defined WITH ACR system before any stub is built.
  Stubbing without a spec creates a fake contract that breaks at integration time.
- **Data scope:** Portal is for Expression of Interest + Press by Number only.
  Press by Name (individual names) continues through ACR directly.
- **Security:** PII for media applicants; quota data is sensitive. Public-facing form
  must be robust against abuse. Concrete plan: email verification + CAPTCHA +
  rate limiting (max 10 submissions per IP per day). No self-service NOC registration —
  accounts are admin-provisioned by the IOC/OCOG to verified NOC coordinators.
- **Localisation:** English-only for v1. French translation in v1.1. This is a hard call.
  206 NOCs means many non-native English speakers, but i18n tooling adds ~2 weeks.
  The August deadline makes English-only the only viable choice for v1.
- **Multi-event data structures:** All tables scoped to `event_id` in the schema.
  No multi-tenant application logic for LA28 — this is a schema-level commitment only.
  LA28 data pre-populates the system. Future events are a data migration, not a code change.

## Premises

1. The MRP is a public-facing intake system + NOC/IOC management layer. It feeds
   structured data into ACR system, which handles the actual accreditation execution.
   The MRP does not make accreditation decisions — it facilitates the process.

2. The IOC is the buyer/stakeholder; NOC coordinators are the day-to-day users during
   the Portal window (Aug-Oct 2026); IOC Media Ops is the oversight/approver.

3. Multi-tenancy should be reflected in data structures (`event_id` everywhere) but not
   in application logic for now. LA28 data pre-populates the system.

4. The most critical integration is ACR system. The portal must export data in a
   format ACR system can ingest, initially via structured export/API stub.

5. April 2026 testing deadline is a hard constraint. Partial ship is not an option.

## Approaches Considered

### Approach A: Standalone Portal, CSV Handoff
Summary: Full portal UI per IOC spec. Data export as structured CSV/Excel for manual
ACR import by OCOG ACR staff.
Effort: M (human: ~4-6 weeks / CC+gstack: ~2-3 days)
Risk: Low (no dependency on ACR API)
Pros: Zero timeline risk from ACR system; ships fastest
Cons: Manual import step remains a human bottleneck; doesn't fully eliminate friction

### Approach B: Full Portal + ACR API Integration (chosen)
Summary: Build the full MRP per spec, with API-first integration to ACR system.
ACR API is stubbed during development; real implementation swapped in when ready.
Effort: L (human: ~10-14 weeks / CC+gstack: ~3-5 days)
Risk: Medium (ACR API stubbed, so development is decoupled; integration risk
deferred to when the real API is ready)
Pros: Eliminates all manual handoffs; real-time quota sync possible; stronger product story
Cons: Integration complexity; ACR API quality/readiness is unknown

### Approach C: MRP as ACR Module
Summary: Build portal as a module inside or tightly coupled to the ACR platform.
Effort: XL (human: 16+ weeks)
Risk: Very High (platform constraints, shared dev environment, public-facing security harder)
Not chosen: Timeline risk is unacceptable; public form security is harder inside an
enterprise accreditation system.

## Recommended Approach

**Approach B: Full portal, API-first with ACR system stubs.**

Build the full portal per the IOC spec. Use an adapter pattern for the ACR system
integration layer:

```
MRP App
  └── AcrAdapter (interface)
        ├── AcrStubClient (dev/test — returns realistic fixture data)
        └── AcrApiClient (prod — real ACR API, swapped in when ready)
```

This means development is never blocked on ACR API readiness. The stub client
returns realistic quota data so all quota enforcement logic can be built and tested
fully. When ACR API is ready, only the `AcrApiClient` class changes.

**Data model foundation:**
Every table is scoped to `event_id`. Example: `media_applications(id, event_id, noc_id, ...)`.
For LA28, a single event record is created and all data references it. Adding future events
is a data operation, not a code change.

## Resolved Decisions (was Open Questions)

These were open; they are now resolved:

1. **ACR API contract:** Sprint 1 task. Define the AcrAdapter interface with
   ACR integration team before writing a single stub. Required fields: `fetchQuota(noc_id, event_id)`,
   `pushOrgData(org[], event_id)`, `getOrgCodes(event_id)`. ACR system must provide a sandbox
   environment before integration testing begins. **Unblocked by:** Ken to schedule API
   design session with ACR integration team this week.

2. **Authentication for NOC coordinators:** Admin-provisioned accounts only (v1).
   The IOC/OCOG maintains the list of authorised NOC contacts (they already have this from
   quota communication). Accounts are created by the OCOG ACR admin, credentials emailed
   to NOC contacts. Auth: email + password, email verification required.
   No SSO for v1 (avoids IOC SSO timeline dependency). Self-service registration in v1.1.

3. **Public form spam/abuse:** Email verification + CAPTCHA (hCaptcha or similar) +
   rate limiting (max 10 submissions per IP per day, 3 per email per day).
   Application form requires: valid email + name + organisation name. Minimal barrier,
   practical limit on bulk abuse.

4. **Document uploads:** Deferred to v1.1. v1 form collects org name, contact details,
   category requested, and a free-text "About your organisation" field. No file uploads.
   This significantly reduces storage and security complexity.

5. **Language/localisation:** English-only for v1. Hard decision to hit August deadline.
   French in v1.1.

6. **Deduplication algorithm (v1):**

   ```
   normalise(s) = lowercase(strip_punctuation(collapse_whitespace(s)))

   IF normalise(org_name) matches AND domain matches → AUTO_DUPLICATE (blocked, IOC notified)
   IF normalise(org_name) matches OR domain matches → FUZZY_FLAG (IOC review queue)
   ELSE → NO_FLAG (proceeds normally)
   ```

   Domain extraction: parse from website URL field (e.g., `apnews.com` from `https://apnews.com/about`).
   Corporate NAT / shared-IP edge case: rate limit is 10 per IP per day with manual review
   queue for legitimate orgs caught by IP limit (flagged by multiple submissions, same org name).
   No ML required for v1. Human IOC review handles ambiguous cases.
   False positive rate: acceptable — IOC reviews the queue daily. False negatives are worse
   (same org accredited through multiple NOCs is a reputational issue for the IOC).

7. **Press by Number IOC approval:** Out-of-band for v1. NOCs download their approved
   applications list as CSV, the OCOG processes via existing workflow. In-system
   approval workflow is v1.1 (launches before Oct 5, 2026 Press by Number deadline).

8. **Undertaking signature:** External process (Adobe Acrobat) continues. Deferred to v1.1.

## System Reliability & Recovery

**ACR API failure modes:**
- `fetchQuota()` unavailable → cache last-known quota in MRP DB, surface staleness warning to IOC dashboard. Do not block NOC operations.
- `pushOrgData()` fails → queue for retry (exponential backoff, max 5 attempts, 24h window). Alert OCOG ACR admin on permanent failure. Log all attempts with payload for manual recovery.
- **Rollback:** MRP is the system of record for Expression of Interest data. ACR integration is write-forward. If bad data is pushed, OCOG ACR corrects it in ACR directly. MRP logs all pushes with timestamps for audit.

**NOC deadline enforcement:**
- Portal hard-closes at midnight Oct 23, 2026 UTC. No new applications accepted.
- Read-only access continues until end of Games for audit/reference.

**Data classification:**
| Field | Classification | Storage |
|-------|---------------|---------|
| Org name, website | Public | MRP DB (encrypted at rest) |
| Contact name, email | PII (Restricted) | MRP DB (encrypted at rest) |
| Quota allocations | Internal | MRP DB |
| Exported to ACR system | Internal | ACR system (their responsibility) |

All PII handled under GDPR Article 6(1)(e) (public task performed by IOC as public authority).
Data controller: IOC. Data processor: OCOG/D.TEC.
Retention: archive until Dec 31, 2030 (2 years post-Games), then purge.
Backup retention: 90 days rolling.
Right-to-be-forgotten: requests routed to IOC Media Ops. IOC asserts legitimate interest for
retention of accreditation records (dispute resolution) — right-to-erasure restricted under
Article 17(3)(b). Legal review required before launch.
