**RESOLVED — 2026-03-29:** Flat model confirmed for v0.09 and v0.1 build. AP-US and AP-France are independent org records — each managed by their respective NOC, linked only by shared domain as a multi-territory flag. Same domain + same NOC = true duplicate (block). Same domain + different NOC = multi-territory flag (allow, IOC informed). Parent-child model deferred; can be added later if IOC requires corporate hierarchy signal.

**What:** Resolve the org identity model before building the dedup logic. Core question: is AP-Japan the same organization as AP-US, or a separate entity? **Why:** This is a foundational data model question. The dedup logic, Common Codes integration, and cross-Games persistence all depend on the answer. Getting it wrong means a migration later. The PROVISIONAL decision is Option A (flat), because: (1) from an accreditation standpoint AP-Japan and AP-US are coded differently in Common Codes, get different quotas, and have different credentials; (2) each NOC manages its own territory independently; (3) cross-territory same-domain is a "multi-territory flag" not a true duplicate. Confirm with IOC.
**Pros:** Flat model is simpler to build and reflects how OCOG ACR codes orgs. Cheap to extend to parent-child later.
**Cons:** Loses the corporate hierarchy signal. Multi-territory same-org must be an explicit relationship, not implied.
**Context:** Provisional dedup rule change: same domain + same NOC = true duplicate (block). Same domain + different NOC = multi-territory flag (allow, IOC informed). This replaces the earlier "same domain across any NOC = possible duplicate" rule. ### [TODO-022] Freelancer data model — hybrid org-of-1 and collective workflow
**What:** Define how freelancers are handled in the data model. Two cases: (A) Independent freelancer — applies as themselves, "org of 1." Single person = single org record. (B) Freelancer collective — a group of freelancers under a shared umbrella (e.g., a photo agency, a content collective). The collective applies for a quota; individual freelancers are listed under it. Design the cleanest UX path for both cases without creating two separate application flows.
**Why:** Freelancers are a meaningful segment of media applicants. The "org of 1" concept (from the March 12 meeting) simplifies the data model by treating everyone as an org. But a freelancer who is part of a collective needs to be associated with that collective, not just with themselves. Getting the model wrong creates dedup problems and breaks the Common Codes org record workflow.
**Pros:** Clean model that handles both cases. Avoids collecting individual PII at EoI stage (name+country for dedup, not passport details).
**Cons:** Adds complexity to the application form (how does a freelancer indicate they're part of a collective vs. independent?). Collective approval and quota allocation path differs from named-org path.
**Context:** Dedup rule for freelancers: name + country (not domain). Independent freelancers get their own org record in Common Codes if quota approved. Collective freelancers are associated with the collective's org record. The form needs to distinguish these two cases cleanly — likely via an applicant type selector early in the flow ("Are you applying as an individual or on behalf of an organization/collective?"). Confirm with IOC OIS whether collectives are a real use case or edge case.
**Effort:** M (design session + ~half day dev once model confirmed) | **Priority:** P2 | **Depends on:** IOC OIS confirmation of freelancer collective use case, TODO-003
