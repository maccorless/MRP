### [TODO-P0-F] Local/host-NOC accreditation extension (USOPC domestic accreds) — gather requirements from Martyn
**What:** USOPC (and potentially other host-country NOCs in future Games) issues a special category of accreditation for local/domestic-only media that does not map cleanly onto the standard E / Es / EP / EPs / ET / EC categories. Today this is USOPC-specific for LA28 because the USA is the host country. We need to capture: what the local accreds are called, what they grant (Games-time access? city-only? venue subset?), what additional EoI fields are required (e.g. local press ID, state/city, domestic publication scope), how the quota works (separate pool? inside or outside NOC E totals?), and how they flow to ACR.
**Design intent — do NOT hard-code to USOPC:** Model this as a generic "local/host-NOC credential process" capability that any NOC can opt into per Games (e.g. a `supports_local_credentials` flag on the NOC record). When enabled, the EoI form surfaces an additional category + extra fields driven by a per-NOC local-credential config. For LA28 only USOPC turns it on. For Brisbane 2032 it would flip to AOC. This keeps the data model clean and reusable and avoids baking a single NOC's rules into core logic.
**Open questions for Martyn:**
1. What are the local accred categories USOPC uses, and are they Games-edition-specific or carried across editions?
2. Do local accreds consume slots from the standard NOC E pool, or is there a separate quota set by IOC/OCOG?
3. Are local accreds ever issued to orgs that would also appear on the main PbN? (Double-credential risk.)
4. Which additional applicant fields must the EoI form collect for a local accred application, and who validates them?
5. Do local accreds flow through the same PbN → OCOG approval → ACR handoff, or do they take a separate path?
6. Are local accreds ever nominated by OCOG rather than by the NOC? (Host-country OCOG may have a direct route.)
**Why:** Currently invisible in our design. Shipping v1 without a home for USOPC's local accreds means USOPC runs this process entirely offline — exactly the Excel/side-channel workflow we're trying to reduce. Even if we can't deliver the full feature by August 24, we need to reserve the schema shape so we don't paint ourselves into a corner.
**Depends on:** Requirements discovery with Martyn (OCOG), then a secondary confirmation with a USOPC contact. The Thursday 2026-04-16 meeting is the opening for this conversation — ask Martyn to surface it and nominate a USOPC contact for follow-up.
**Priority:** P0 (for discovery on Thursday). Implementation priority will set after requirements land — likely P1 for v1.1.

### [TODO-012] Org identity model — foundational data model decision required
**RESOLVED — 2026-03-29:** Flat model confirmed for v0.09 and v0.1 build. AP-US and AP-France are independent org records — each managed by their respective NOC, linked only by shared domain as a multi-territory flag. Same domain + same NOC = true duplicate (block). Same domain + different NOC = multi-territory flag (allow, IOC informed). Parent-child model deferred; can be added later if IOC requires corporate hierarchy signal.

**What:** Resolve the org identity model before building the dedup logic. Core question: is AP-Japan the same organization as AP-US, or a separate entity? Two options: (A) Flat model — AP-Japan and AP-US are separate org records, each managed independently by their NOC, linked only by shared domain; (B) Parent-child model — AP is a root entity, AP-Japan and AP-US are territory-scoped child records.
**Why:** This is a foundational data model question. The dedup logic, Common Codes integration, and cross-Games persistence all depend on the answer. Getting it wrong means a migration later. **Cons:** Loses the corporate hierarchy signal. Multi-territory same-org must be an explicit relationship, not implied.
